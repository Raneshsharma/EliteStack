## P1.1: Onboarding

**Primary Owner:** `Product Manager Agent`
**Support:** `User Flow Agent`, `UX Lead`, `Wireframe Agent`, `Frontend Tech Lead`, `UI Component Agent`, `Functional Test Agent`
**Watchers:** `Project Manager Agent`

### 1. Problem Statement

**Who has this problem?** Every new user who registers for an account and lands in the product for the first time — with no resume data, no context, and no idea what to do next.

**What is the problem?** After registration, users often feel lost. They see an empty dashboard with no guidance, don't know what to click first, and may abandon the product before creating their first resume. A poor onboarding experience kills retention before the product has a chance to prove its value.

**Why does it matter?** First impressions are critical. The onboarding experience determines whether users understand the product's value quickly ("aha moment") or get confused and leave. Reducing time-to-first-resume is directly tied to onboarding quality. Every step of friction in onboarding loses users.

**Current workaround (if any):** Users figure it out by exploring the interface, clicking around, or giving up and leaving. Without guided onboarding, the product relies on users being highly motivated — which most aren't.

---

### 2. Goal / Objective

**User Goal:** Feel welcomed and guided from the moment I register. Understand what the product does and how to create my first resume in under 2 minutes — without guessing.

**Business Goal:** Maximize the "time-to-first-resume" rate. Get users to their "aha moment" (seeing a complete resume preview) within 5 minutes of registration. Reduce early abandonment by 50%.

**Success Definition:** Over 70% of new users who complete registration also complete their first resume within the same session. Zero users see an empty dashboard with no call-to-action.

---

### 3. Scope

**In-Scope:**
- **Welcome page:** Shown immediately after first login (post-registration). Personalized with user's name. Brief product intro (3–4 lines max).
- **First-resume CTA:** Prominent "Create your first resume" button on welcome page and empty dashboard state.
- **Skip option:** "Skip for now" link for returning users who have seen the welcome before.
- **Progress indicator:** Step tracker showing users where they are in the first-resume creation flow.
- **Empty state design:** Empty dashboard has a compelling visual + "Create your first resume" CTA, not just blank space.
- **Onboarding checklist:** Optional checklist widget showing: "Add your info → Add experience → Preview → Export" with checkmarks.
- **Feature discovery tooltips:** Subtle, dismissible tooltips on first visit to key UI elements (e.g., "Tip: Your resume auto-saves as you type").

**Out-of-Scope:**
- Interactive product tour (walk-through with multiple steps)
- Onboarding analytics / in-app messaging platform
- Email drip campaigns for onboarding
- Gamification (points, badges, streaks)
- Video tutorials or modal walkthroughs
- Custom onboarding flows per subscription tier

---

### 4. User Stories

**US-P1.1-A:** As a **newly registered user**, I want to **see a welcome screen after logging in**, so that I understand **what the product does and what to do next**.

**US-P1.1-B:** As a **newly registered user**, I want to **create my first resume with one click** from the welcome screen, so that I can **start immediately without exploring the interface**.

**US-P1.1-C:** As a **returning user** who has seen the onboarding, I want to **skip the welcome screen**, so that I can **go directly to my dashboard**.

**US-P1.1-D:** As a **newly registered user**, I want to **see a clear checklist** of what to do next, so that I **don't miss any important steps** when building my first resume.

**US-P1.1-E:** As a **newly registered user**, I want to **see an empty state on the dashboard** that prompts me to create a resume, so that I know **the product is working even with no data**.

**US-P1.1-F:** As a **new user**, I want to **see subtle tooltips** on key features the first time I use them, so that I **discover functionality without needing external documentation**.

---

### 5. User Flow / Experience

**First-Time User Flow:**
1. User completes registration → auto-logged-in
2. Redirected to `/onboarding/welcome/` (first time only)
3. Welcome page shows: "Welcome, [FirstName]!", brief value prop (2 sentences), "Create your first resume" primary CTA, "Skip for now" secondary link
4. User clicks "Create your first resume" → system creates draft resume → redirects to `/resume/<id>/edit/`
5. Dashboard shows onboarding checklist sidebar: "Add your info ✓", "Add experience ○", "Preview ○", "Export ○"
6. Checklist items check off as sections are filled
7. On first successful export, checklist shows completion state and onboarding is marked complete

**Returning User (Post-Onboarding) Flow:**
1. User logs in with existing account
2. Redirected to `/dashboard/` — no welcome page shown
3. Dashboard shows normal resume list (or empty state if no resumes)

**Empty State Dashboard (No Resumes):**
1. Dashboard shows: friendly illustration/icon, headline "You don't have any resumes yet", subtext "Create your first resume and take the next step in your career", primary "Create your first resume" button, secondary "Learn more" link
2. This empty state is shown whether or not the user completed onboarding

**Onboarding Checklist Widget:**
1. Persistent sidebar or top-of-page widget on builder page for users with < 1 complete resume
2. Shows: checkmark for completed sections, circle for incomplete sections
3. Tooltips on hover explain what each section needs
4. Dismissible — user can collapse the widget

### 10. Technical Notes

- **Welcome Page Route:** `/onboarding/welcome/` — only rendered if `user.profile.onboarding_completed == False`.
- **Onboarding State:** Add `onboarding_completed` boolean field to `UserProfile` model. Set to `True` when user completes their first resume or manually skips.
- **Onboarding Checklist:** Conditionally rendered in the builder template if `user.resumes.filter(status='complete').count() == 0`. JavaScript-driven check-off as sections are filled.
- **Tooltips:** Use a lightweight library like `tippy.js` or vanilla CSS tooltips. Content stored in a JSON config object in the template.
- **Skip Logic:** A link to `/onboarding/skip/` sets `onboarding_completed = True` and redirects to dashboard.
- **Empty State Component:** Create a reusable `empty_state.html` template partial used by both onboarding CTA and dashboard empty state.
- **First Name:** Access via `request.user.first_name`. Fall back to "there" if blank.

## P1.2: Dashboard

**Primary Owners:** `Frontend Tech Lead`, `Backend Tech Lead`
**Support:** `Dashboard Architect Agent`, `Widget Agent`, `State Management Agent`, `UI Component Agent`, `Functional Test Agent`
**Watchers:** `Product Manager Agent`, `Project Manager Agent`

### 1. Problem Statement

**Who has this problem?** Every logged-in user who needs to quickly find, manage, and act on their resumes — without digging through menus or remembering URLs.

**What is the problem?** Without a central hub, users have no clear landing page after login. They may not know where to go to create a new resume, view an existing one, or understand their overall resume activity. The dashboard is the command center for the entire product.

**Why does it matter?** The dashboard sets the tone for the entire product experience. A well-designed dashboard makes users feel organized and in control. A poor one creates confusion and makes the product feel unfinished. It also determines how quickly users can take action — which directly impacts daily active usage.

**Current workaround (if any):** Without a dedicated dashboard, users might land on a profile page, a settings page, or nowhere useful at all. They have no quick overview of their resume health.

---

### 2. Goal / Objective

**User Goal:** Land on a single page after login that shows all my resumes, quick actions, and a clear sense of where I am in my job search.

**Business Goal:** Increase daily active usage by giving users a reason to return. Surface the most important actions (create, view, edit) front and center. Build a foundation for future widgets (resume stats, activity feed, subscription status).

**Success Definition:** The dashboard loads in under 1 second. Users can find any of their resumes within 3 seconds. The "Create New Resume" CTA is visible without scrolling. Resume statistics are accurate and up-to-date.

---

- **Quick action buttons:** "Create New Resume" (primary), plus contextual actions per resume card
- **Resume statistics widget:** Total resumes count, completed resumes count, draft resumes count, last edited date
- **Recent activity feed:** Last 5 actions (created, edited, exported, duplicated) across all resumes
- **Empty state:** Friendly prompt to create first resume when user has no resumes
- **Navigation header:** Consistent top navigation with links to Dashboard, Resumes, Settings, Logout
- **Breadcrumb / page title:** Clear indication of current location ("My Dashboard")

**Out-of-Scope:**
- Subscription status widget (future — tied to P3 monetization)
- Analytics dashboard (views, downloads, apply counts — future)
- Customizable dashboard layout (drag-and-drop widgets)
- Email notification preferences widget
- Job application tracker widget
- Team/shared resume management
- Dark mode toggle

### 10. Technical Notes

- **URL:** `GET /dashboard/` — requires authentication. Unauthenticated requests redirect to login.
- **Template:** `dashboard.html` — extends base template. Includes: `statistics_widget.html`, `resume_card.html`, `activity_feed.html`, `empty_state.html` partials.
- **Statistics Query:** `Resume.objects.filter(user=request.user).aggregate(total=Count('id'), complete=Count(Case(status='complete')), draft=Count(Case(status='draft'))`. Cache for 60 seconds.
- **Recent Activity:** Store activity log in a `ResumeActivityLog` model or query recent `updated_at` changes across resumes. Cap at 5 entries.
- **Resume List Query:** `Resume.objects.filter(user=request.user).order_by('-updated_at')`. Use `select_related` on user. Paginate at 20.
- **Performance:** Dashboard should make no more than 3 database queries (resume list, stats, activity). Use `prefetch_related` for related entries if needed.
- **Mobile Layout:** Cards stack vertically. Statistics widget moves above resume list. Activity feed moves below resume list.
- **Redirect on Login:** Set `LOGIN_REDIRECT_URL = '/dashboard/'` in `settings.py`.

## P1.3: Resume Templates

**Primary Owner:** `UI Component Agent`
**Support:** `Template Architect Agent`, `CSS/Animation Agent`, `Frontend Tech Lead`, `QA Lead`, `Functional Test Agent`
**Watchers:** `Product Manager Agent`, `System Architect`

### 1. Problem Statement

**Who has this problem?** Every resume builder user who wants their resume to look professional and stand out — but doesn't have design skills or the time to customize formatting.

**What is the problem?** Users have different aesthetic preferences and different industries expect different resume formats. A tech startup might prefer a modern, clean resume; a law firm might expect a traditional, serif-font resume. Without template options, the product serves one style only — alienating users whose needs don't match that style.

**Why does it matter?** Resume appearance directly affects how recruiters perceive candidates. A resume that looks outdated or mismatched to the industry signals poor judgment. Offering template choices increases perceived value and differentiates the product from competitors with one-size-fits-all builders.

**Current workaround (if any):** Users who want a different style must use a different tool entirely — paying for Resume.io, Zety, or similar services, or manually recreating formatting in Word.

---

### 2. Goal / Objective

**User Goal:** Choose from multiple professionally designed resume templates that match my industry, personality, and target role — and switch between them without losing my content.

**Business Goal:** Increase resume completion and export rates by giving users aesthetic choice. Differentiate the product from single-template competitors. Build the template system architecture to support future premium templates.

**Success Definition:** Users can view 3 distinct template previews before selecting one. Switching templates preserves all resume content (data is never lost). The selected template persists across sessions and is used for both preview and export.

---

### 3. Scope

**In-Scope:**
- **Template gallery:** A dedicated page (`/templates/`) showing all available template previews with names and brief descriptions
- **Three template styles at launch:**
  - **Classic:** Traditional, serif fonts, standard margins — conservative, widely accepted
  - **Modern:** Sans-serif, strong visual hierarchy, accent colors — clean and contemporary
  - **Minimal:** Maximum whitespace, simple typography — elegant and ATS-friendly
- **Template preview:** Each template shown as a rendered sample resume in the gallery
- **Template selection:** User selects a template — it becomes the default for that resume
- **Template switching:** User can switch templates at any time from the builder — all content preserved
- **Default template:** Classic is the default for new resumes; user can change their default in Settings
- **Per-resume template:** Each resume remembers its own selected template independently

**Out-of-Scope:**
- Additional template styles beyond the 3 defined (P3 may add more)
- Customizable template colors, fonts, or layouts
- Template bundling (grouping templates by industry)
- Template ratings or user reviews
- Template favorites / bookmarking
- Premium vs. free template tiers
- Template upload by users (upload your own HTML/CSS)
- AI-generated custom templates

---

### 4. User Stories

**US-P1.3-A:** As a **logged-in user**, I want to **browse a gallery of available templates** with preview thumbnails, so that I can **choose a style before I commit to one**.

**US-P1.3-B:** As a **logged-in user**, I want to **switch my resume's template** at any time from the builder, so that I can **try different looks without losing my content**.

**US-P1.3-C:** As a **logged-in user**, I want to **set a default template** that applies to all new resumes, so that I **don't have to select the same template every time**.

**US-P1.3-D:** As a **logged-in user**, I want each resume to **remember its own selected template** independently, so that I can **have different styled resumes for different purposes**.

**US-P1.3-E:** As a **logged-in user**, I want to **see a rendered preview** of each template before selecting it, so that I can **make an informed choice**.

**US-P1.3-F:** As a **user who cares about ATS**, I want to **know which templates are most ATS-friendly**, so that I can **choose a template that won't hurt my application**.

---

### 5. User Flow / Experience

**Template Gallery Flow:**
1. User navigates to `/templates/` or clicks "Templates" in the nav
2. Gallery page displays 3 template cards in a grid, each showing:
   - Template name and brief description (e.g., "Classic: Traditional, professional, widely accepted")
   - A rendered sample resume preview (scaled-down thumbnail)
   - "Use Template" button
3. User clicks "Use Template" on a template card
4. If user has an active resume in the builder: template switches immediately, preview re-renders
5. If no active builder session: user is redirected to create a new resume with that template pre-selected

**Template Switching Flow (Builder):**
1. User is in the resume builder (`/resume/<id>/edit/`)
2. User clicks a template switcher (dropdown or tab) above the preview panel
3. Options show: "Classic", "Modern", "Minimal" with small preview thumbnails
4. User selects a different template
5. Preview re-renders immediately with the new template
6. Selected template is saved to the resume record
7. Toast: "Template updated"

**Default Template Setting Flow:**
1. User navigates to Settings (`/settings/`)
2. "Default Resume Template" section shows 3 template options with previews
3. User selects a template
4. Selection is saved to `UserProfile.default_template`
5. All future new resumes use this template by default

### 10. Technical Notes

- **Template Model:** Store template references as a `CharField` with choices: `('classic', 'Modern', 'minimal')` on the `Resume` model. Default in `UserProfile`.
- **Template Files:** Each template has its own directory: `templates/resumes/classic/`, `templates/resumes/modern/`, `templates/resumes/minimal/`. Each contains `resume.html` and `resume.css`.
- **Base Template:** All templates extend `templates/resumes/base.html` which defines the shared HTML structure (header, sections, footer).
- **Template Switching (Frontend):** JavaScript `fetch()` to `PATCH /resume/<id>/` with `{template: 'modern'}` on template selection. Preview re-renders with new template CSS class on the preview container.
- **Gallery Template:** `templates/resumes/gallery.html` — renders each template with a sample resume context for thumbnails.
- **CSS Isolation:** Each template's CSS should be scoped to prevent bleed. Use CSS custom properties (`--template-primary-color`) for template-specific theming within a shared base.
- **Preview Thumbnail Generation:** In the gallery, the thumbnail is generated by rendering the template with sample data in a small iframe or scaled div (`transform: scale(0.3)` with `transform-origin: top left`).
- **Settings Integration:** `UserProfile.default_template` CharField. Settings page reads and writes this field.

## P1.4: Minimal Settings

**Primary Owner:** `Frontend Tech Lead`
**Support:** `Feature Definition Agent`, `UI Component Agent`, `Form/Validation Agent`, `Auth Agent`, `Functional Test Agent`
**Watchers:** `Product Manager Agent`, `Project Manager Agent`

### 1. Problem Statement

**Who has this problem?** Every logged-in user who needs to update their account information, change their password, set preferences, or manage their account.

**What is the problem?** Without a settings page, users cannot change their password, update their email, or configure preferences. They also have no way to manage their account (delete account, update notification preferences) — which erodes trust and creates a dead-end experience.

**Why does it matter?** Settings is a basic expectation for any authenticated product. Users who want to change their email, tighten security with a new password, or delete their account will leave frustrated if these options don't exist. It also creates a support burden.

**Current workaround (if any):** Users contact support to change passwords or delete accounts — an expensive and frustrating path for everyone.

---

### 2. Goal / Objective

**User Goal:** Manage my account, update my profile, change my password, and configure my preferences in one place — without needing support.

**Business Goal:** Reduce support tickets related to account management. Enable self-service for common account operations. Store user preferences that drive personalization across the product.

**Success Definition:** Users can update their display name, email, and password without errors. Users can delete their account with confirmation. Preferences (default template, paper size) are saved and applied correctly across the product.

---

### 3. Scope

**In-Scope:**
- **Profile settings:** Update display name, email address
- **Password change:** Change password with current password verification
- **Account deletion:** Delete account with confirmation and data removal
- **Preferences:**
  - Default resume template (Classic / Modern / Minimal)
  - Default paper size (US Letter / A4)
  - Email preferences (optional: newsletter, product updates — checkbox)
- **Account info display:** Show username, email, account creation date, resume count

**Out-of-Scope:**
- Profile photo upload
- Linked account management (OAuth connections — future)
- Two-factor authentication settings
- Login activity / session management
- Notification preferences beyond email opt-out
- Referral program settings
- Billing / subscription management (P3)
- API key management
- Language / locale preferences
- Timezone settings

---

### 4. User Stories

**US-P1.4-A:** As a **logged-in user**, I want to **update my display name and email** in settings, so that I can **keep my profile accurate**.

**US-P1.4-B:** As a **logged-in user**, I want to **change my password**, so that I can **maintain account security**.

**US-P1.4-C:** As a **logged-in user**, I want to **set my default resume template**, so that **new resumes I create use my preferred style automatically**.

**US-P1.4-D:** As a **logged-in user**, I want to **set my default paper size**, so that **new resumes I create use my preferred paper size automatically**.

**US-P1.4-E:** As a **logged-in user**, I want to **opt out of marketing emails**, so that I can **receive only transactional emails** (account-related, resume exports).

**US-P1.4-F:** As a **logged-in user**, I want to **delete my account**, so that I can **permanently remove my data from the platform**.

**US-P1.4-G:** As a **logged-in user**, I want to **see my account information** (username, email, account age), so that I can **verify my account details**.

### 5. User Flow / Experience

**Settings Page Flow:**
1. User clicks "Settings" in the navigation bar
2. Navigates to `/settings/`
3. Settings page loads with tabbed or single-page layout:
   - **Profile tab:** Display name field, email field, "Save Changes" button
   - **Security tab:** Current password, new password, confirm new password, "Update Password" button
   - **Preferences tab:** Default template selector (3 cards), paper size toggle (US Letter / A4), email opt-out checkbox
   - **Account tab:** Account info display, "Delete Account" button (danger zone)
4. Each section saves independently with its own "Save" button
5. On save: toast notification "Settings saved"

**Password Change Flow:**
1. User navigates to Security tab
2. Fills: Current password, New password, Confirm new password
3. Submits
4. System validates: current password matches, new password meets complexity, passwords match
5. On success: password updated, session preserved, success toast
6. On error: inline field errors

**Account Deletion Flow:**
1. User clicks "Delete Account" in Account tab
2. System shows a multi-step confirmation:
   - Step 1: "Are you sure? This will permanently delete all your data including [N] resumes."
   - Step 2: "To confirm, type your email address: [user@email.com]"
   - Step 3: "Type DELETE to confirm"
3. User completes confirmation and clicks "Permanently Delete"
4. All user data is deleted from the database
5. User is logged out and redirected to the homepage with a "Account deleted" message

**Preference Change Flow:**
1. User navigates to Preferences tab
2. Selects default template: clicks on a template card (highlighted when selected)
3. Toggles paper size: radio buttons for US Letter / A4
4. Toggles email opt-out: checkbox
5. Clicks "Save Preferences"
6. Preferences saved to `UserProfile`
7. New resumes created after this use the new defaults

### 7. Edge Cases & Error Handling

| Scenario | Expected Behavior |
|----------|-------------------|
| User enters invalid new password (too short, too common) | Inline error below field with Django validator message |
| User enters wrong current password | Inline error: "Your current password was entered incorrectly." |

| User deletes account while another tab is active | Other tab shows "Session expired" on next interaction |
| User clicks "Delete Account" by mistake | Multi-step confirmation prevents accidental deletion |

| User tries to access `/settings/` when not logged in | Redirect to login page |

### 10. Technical Notes

- **URL:** `GET/POST /settings/` — class-based view with multiple form handling.
- **Model:** Extend `UserProfile` model with fields: `default_template` (CharField), `default_paper_size` (CharField), `email_opt_out` (BooleanField), `onboarding_completed` (BooleanField).
- **View Strategy:** Use a single `SettingsView` with multiple Django `ModelForm` or `Form` instances, each handling one section (profile, security, preferences). Use `form_valid()` to save each form independently.
- **Password Change:** Use Django's `PasswordChangeView` (`django.contrib.auth.views.PasswordChangeView`) with custom template.
- **Account Deletion:** Custom view with `POST /settings/delete-account/`. Requires multi-step token-based confirmation (step 1: email input, step 2: "DELETE" token). Use Django's `user.delete()` with cascade. Log deletion for audit purposes.
- **Template Preference:** Write selected template to `request.user.profile.default_template`. Read in the builder and resume creation views.
- **Unsaved Changes Guard:** JavaScript `beforeunload` event listener on settings page. Set a flag when any form field changes. Trigger browser confirmation on navigate away.
- **CSRF:** All forms use `{% csrf_token %}`.
- **Toasts:** Use a reusable toast notification component (JavaScript-driven, CSS-styled).
