  P0.1: Authentication

  Primary Owner: Backend Tech Lead
  Support: Auth Agent, Security Test Agent, Frontend Tech Lead, Form/Validation Agent, Database Agent
  Watchers: Product Manager Agent, Project Manager Agent

  1. Problem Statement

  Who has this problem? Every user who wants to create, save, and manage their resumes.

  What is the problem? Without authentication, users have no way to securely store their resume data, access it across devices, or manage
  multiple resumes under one account. Any anonymous session-based approach loses data on browser close and cannot support the multi-resume
  management required in P0.3.

  Why does it matter? Authentication is the foundation of the entire product. Without it, nothing else works. Users cannot persist data,
  employers cannot return to edit drafts, and the subscription model (required for monetization) has nowhere to attach billing.

  Current workaround (if any): Users would need to manually manage resume files locally (Word docs, PDFs), share them via email, and track
  versions manually — which is exactly what the product aims to eliminate.

  ---
  2. Goal / Objective

  User Goal: Create an account in under 2 minutes, log in securely, and access their resume data anywhere.

  Business Goal: Establish a user identity that supports multi-resume management, future subscription billing, and personalized AI features.

  Success Definition: A new user can sign up with email/password, receive confirmation of account creation, log in, and log out — all without
  error messages or confusion. Every protected page redirects unauthenticated users to login. Every authenticated action is scoped to the
  correct user.

  ---
  3. Scope

  In-Scope:
  - User registration (username, email, password)
  - User login (email/username + password)
  - User logout (session termination)
  - Password change (authenticated users)
  - Password reset via email link (Django auth built-in)
  - Session management (secure, HttpOnly cookies)
  - Protected route middleware (redirect unauthenticated users)
  - User profile: display name, email

  Out-of-Scope:
  - Email verification (can add in P1 if needed)
  - OAuth / social login (Google, LinkedIn)
  - Two-factor authentication
  - Login with magic links
  - CAPTCHA on registration
  - Brute-force protection / rate limiting
  - Custom user model beyond Django's AbstractUser
  - User roles (admin vs. regular — handled by Django staff flag)

  ---
  4. User Stories

  US-P0.1-A: As a new visitor, I want to create an account with my email and a password, so that I can securely store and access my resumes from
   any device.

  US-P0.1-B: As a returning user, I want to log in with my email and password, so that I can access my existing resumes.

  US-P0.1-C: As a logged-in user, I want to log out, so that I can ensure my account is secure on shared devices.

  US-P0.1-D: As a logged-in user, I want to change my password, so that I can maintain account security if I suspect my password has been
  compromised.

  US-P0.1-E: As a logged-in user, I want to reset my password via email, so that I can regain access if I forget my password.

  US-P0.1-F: As a guest (unauthenticated user), I want to be redirected to a login page when I try to access protected routes, so that I
  understand I need to be signed in.

  ---
  5. User Flow / Experience

  Sign Up Flow:
  1. User lands on /accounts/signup/
  2. Form displays: Username, Email, Password, Password (confirm)
  3. User fills form and submits
  4. System validates: username uniqueness, email uniqueness + format, password complexity, passwords match
  5. On success: Account created, user auto-logged-in, redirected to dashboard
  6. On error: Inline error messages below each field, form retained

  Login Flow:
  1. User lands on /accounts/login/
  2. Form displays: Email/Username, Password
  3. User fills form and submits
  4. System validates credentials
  5. On success: Session created, redirected to /dashboard/ or intended URL (?next=)
  6. On error: Generic "Invalid username/email or password" message (do NOT reveal which field)

  Logout Flow:
  1. User clicks "Logout" in navigation
  2. Session destroyed server-side
  3. User redirected to /accounts/login/ with "You have been logged out" message

  Password Reset Flow:
  1. User clicks "Forgot password?" on login page
  2. Page displays: Email field
  3. User submits email
  4. If email exists: System sends reset link (console in dev, email in prod)
  5. User clicks link in email → lands on reset form
  6. User sets new password → redirected to login with success message

  Password Change Flow:
  1. Authenticated user navigates to /accounts/password_change/
  2. Form: Current password, New password, Confirm new password
  3. On success: Confirmation message, session preserved
  4. On error: Inline field errors

  ---
  6. Acceptance Criteria

  - AC-P0.1-01: A user can register with username, email, and password. Upon registration, they are automatically logged in and redirected to
  /dashboard/.
  - AC-P0.1-02: A user can log in with their email/username and password. On success, redirected to /dashboard/ or the URL they originally tried
   to access (?next= param).
  - AC-P0.1-03: A user can log out. Their session is destroyed and they are redirected to the login page with a "Logged out" message.
  - AC-P0.1-04: Unauthenticated users accessing /dashboard/, /resume/, /resume/new/, or any protected URL are redirected to
  /accounts/login/?next=<original-url>.
  - AC-P0.1-05: After logging in from a redirect, the user is sent to the originally requested URL.
  - AC-P0.1-06: Password reset flow sends an email with a valid reset link. The link allows setting a new password.
  - AC-P0.1-07: Password change (authenticated) works with current password validation.
  - AC-P0.1-08: Duplicate username or email during registration shows an inline error without revealing whether the username or email
  specifically is taken.
  - AC-P0.1-09: Invalid login credentials show a generic "Invalid username/email or password" message.
  - AC-P0.1-10: All auth forms use CSRF protection via Django's template tags.

  9. Dependencies & Assumptions

  Dependencies:
  - Django 4.2+ with django.contrib.auth and django.contrib.messages
  - django-crispy-forms or Django's built-in form rendering
  - Email backend configured (console backend for dev, SMTP for production)
  - PostgreSQL database for user persistence

  Assumptions:

  - Deployment environment supports HTTPS

  Blockers if missing:

  - No email backend → password reset cannot send links

  ---
  10. Technical Notes

  - Use Django's built-in django.contrib.auth.views (LoginView, LogoutView, PasswordResetView, PasswordResetConfirmView, PasswordChangeView)
  with custom templates.
  - Use django.contrib.auth.decorators.login_required on all protected views.
  - Configure LOGIN_URL, LOGIN_REDIRECT_URL, LOGOUT_REDIRECT_URL in settings.py.
  - Use django.contrib.messages for flash success/warning messages.
  - Store sessions in the database: SESSION_ENGINE = 'django.contrib.sessions.backends.db'.
  - Production settings: SESSION_COOKIE_SECURE = True, SESSION_COOKIE_HTTPONLY = True, SESSION_COOKIE_AGE = 1209600 (2 weeks).
  - Use {% csrf_token %} in all auth forms.
  - Override default auth templates under templates/registration/.

  P0.2: Resume Builder

  Primary Owners: Frontend Tech Lead, Backend Tech Lead
  Support: Feature Definition Agent, User Flow Agent, Wireframe Agent, Backend Architect Agent, Database Agent, API Agent, UI Component Agent,
  Form/Validation Agent, QA Lead, Functional Test Agent, Edge Case Agent
  Watchers: Product Manager Agent, System Architect, Project Manager Agent

  1. Problem Statement

  Who has this problem? Every job seeker who needs to create a professional resume — especially college students, fresh graduates, and career
  switchers who have limited experience with resume formatting and ATS requirements.

  What is the problem? Writing a resume from scratch in Word or Google Docs is time-consuming, error-prone, and frustrating. Users struggle with
   knowing what sections to include, formatting consistently, writing impactful bullet points, and deciding what information to include vs.
  omit.

  Why does it matter? The resume is the primary tool job seekers use to present themselves to employers. A poorly structured or formatted resume
   directly reduces interview chances. Users who cannot create a quality resume quickly lose confidence and may give up on the job search
  entirely.

  Current workaround (if any): Manual creation in Word/Google Docs, copying from templates, using free builders with ads and watermarks, or
  paying for professional resume writing services ($100–$500+).

  ---
  2. Goal / Objective

  User Goal: Create a complete, professional-looking resume in under 15 minutes by filling in guided, structured forms — no design skills
  needed.

  Business Goal: Reduce the time-to-first-resume to under 15 minutes. Maximize completion rate. Build a data structure that supports future AI
  enhancement.

  Success Definition: A user can fill out a guided multi-section form, see a live preview, and save a complete resume — all without technical
  confusion or data loss. Auto-save prevents data loss.

  ---
  3. Scope

  In-Scope:
  - Personal Info Section: Full name, professional title/headline, email, phone, city/state/country, LinkedIn URL, portfolio/website URL,
  professional summary
  - Education Section: Multiple entries — school name, degree type (dropdown), field of study, start date, end date (or "Present"), GPA
  (optional), description (optional). Add/edit/delete entries.
  - Experience Section: Multiple entries — company name, job title, location, start date, end date (or "Present"), description (plain text).
  Collapsible entries.
  - Projects Section: Multiple entries — project name, role, URL (optional), date range, description. Add/edit/delete entries.
  - Skills Section: Multiple entries — skill name, proficiency level (Beginner / Intermediate / Advanced / Expert). Display as tags/chips.
  - Auto-save: Resume data saves automatically every 30 seconds when changes are detected.
  - Form validation: Required fields, email format, URL format, date logic (end date >= start date).
  - Progress indication: Section completion indicators ("3 of 5 sections complete").

  Out-of-Scope:
  - AI-generated bullet points or descriptions
  - Job description input for ATS matching
  - Resume scoring or feedback
  - Import from LinkedIn or PDF upload
  - Custom section creation by users
  - Rich text editor (plain text only at MVP)
  - Image upload (profile photo)
  - Cover letter builder
  - Multi-column layouts in the builder form

  ---
  4. User Stories

  US-P0.2-A: As a logged-in user, I want to create a new resume by filling in a guided form, so that I can build a professional resume without
  design or formatting skills.

  US-P0.2-B: As a logged-in user, I want to add multiple education entries, so that I can include all relevant academic history.

  US-P0.2-C: As a logged-in user, I want to add multiple work experience entries with dates and descriptions, so that I can show my professional
   history in detail.

  US-P0.2-D: As a logged-in user, I want to add project entries with links, so that I can showcase portfolio or personal projects relevant to my
   target role.

  US-P0.2-E: As a logged-in user, I want to list my skills with proficiency levels, so that I can quickly communicate my technical and soft
  skills.

  US-P0.2-F: As a logged-in user, I want to see a live preview of my resume as I type, so that I can immediately see how it will look when
  finished.

  US-P0.2-G: As a logged-in user, I want my resume to auto-save while I work, so that I never lose my progress if my browser closes or I
  navigate away.

  US-P0.2-H: As a logged-in user, I want to see clear progress indicators for each section, so that I know which sections I still need to fill
  in.

  US-P0.2-I: As a logged-in user, I want to validate my inputs (email, URL, dates), so that I cannot accidentally create an invalid resume.

  ---
  5. User Flow / Experience

  Resume Creation Flow:
  1. User clicks "Create New Resume" from dashboard
  2. System creates a new Resume record (status: draft) → redirects to /resume/<id>/edit/
  3. Builder page loads with left panel (form) and right panel (live preview)
  4. Each section: title with completion indicator, form fields with labels and placeholders, "Add Entry" button for multi-entry sections,
  inline validation errors
  5. User fills Personal Info → auto-saves → preview updates
  6. User clicks "Add Education" → inline form row appears → fills → auto-saves
  7. Repeat for Experience, Projects, Skills
  8. Progress bar at top: "X of 5 sections complete"
  9. User clicks "Save & Preview" → validation runs → if complete, status = "complete"
  10. User clicks "Back to Dashboard" → resume appears in dashboard list

  Inline Entry Management:
  - Each multi-entry section shows a list of existing entries
  - Each entry has: Edit button, Delete button (with confirmation dialog)
  - Empty state: "No entries yet. Click 'Add [Section]' to get started."

  Auto-Save Behavior:
  - Debounced: 30-second timer after last keystroke
  - On save: Toast "Saved" (2s) or subtle "Saving..." indicator
  - On save failure: Red toast "Save failed. Retrying..." with automatic retry

  - AC-P0.2-06: Skills section allows adding, editing, and deleting multiple skills with proficiency levels.
  - AC-P0.2-07: Resume preview updates in real-time (within 1 second) when any form field changes.
  - AC-P0.2-08: Form auto-saves every 30 seconds when changes are detected, without requiring user action.
  - AC-P0.2-09: Progress indicator shows the number of sections completed out of 5.
  - AC-P0.2-10: Email field validates email format with inline error.
  - AC-P0.2-11: URL fields validate URL format with inline error.
  - AC-P0.2-12: End date must be >= start date with inline error if violated.
  - AC-P0.2-13: Required fields cannot be left blank.
  - AC-P0.2-14: User can navigate directly to /resume/<id>/edit/ and see their previously saved data.
  - AC-P0.2-15: Resume status is "complete" when Personal Info is filled, "draft" otherwise.

  10. Technical Notes

  - Two-panel layout: Left panel = form (scrollable), Right panel = live preview (sticky). On mobile, tabs to switch between form and preview.
  - Auto-save: JavaScript setTimeout debounced on input events. Send PATCH /api/resumes/<id>/ with changed fields as JSON. Update preview with
  returned data.
  - Preview rendering: Same Django template used for PDF export. In builder, renders into a <div id="preview">. On every change, re-render.
  - Dynamic form entries: JavaScript to add/remove form entry HTML fragments. Each entry has a hidden id field for existing entries.
  - Date handling: Store as DateField. Display as "MMM YYYY". "Present" stored as null with is_current=True.
  - URL fields: Use Django's URLField with verify_exists=False.
  - Rich text: Plain text with white-space: pre-wrap. No WYSIWYG editor at MVP.
  - Character limits: Summary: 500 chars; descriptions: 2000 chars each. Enforced in both frontend and backend.

  P0.3: Resume Management

  Primary Owner: Backend Tech Lead
  Support: Database Agent, API Agent, Frontend Tech Lead, State Management Agent, Functional Test Agent, Integration Test Agent
  Watchers: Project Manager Agent, System Architect

  1. Problem Statement

  Who has this problem? Every user who creates more than one resume — which is most users. A software engineer may need different resumes for
  "Backend Engineer" vs. "Engineering Manager" roles. A student may need separate resumes for tech vs. consulting applications.

  What is the problem? Without centralized resume management, users juggle multiple Word/PDF files, lose track of which version is the latest,
  and risk sending the wrong resume to an employer. There is no way to duplicate a previous resume as a starting point.

  Why does it matter? Users need flexibility to maintain multiple tailored resumes for different job types. The ability to quickly duplicate and
   modify a resume dramatically reduces friction for job applications.

  Current workaround (if any): Users maintain multiple files named Resume_v1_FINAL_Real.docx, Resume_consulting_v3.docx — a chaotic and
  error-prone system.

  ---
  2. Goal / Objective

  User Goal: Create, view, edit, organize, duplicate, and delete my resumes — with clear visibility into which resume is my primary/default.

  Business Goal: Support a minimum of 5 resumes per user. Enable the duplicate feature to drive re-engagement. Build the foundation for future
  subscription tiers.

  Success Definition: A logged-in user can perform all CRUD operations on their resumes without error. They can identify their default resume at
   a glance. All operations are scoped exclusively to the logged-in user.

  ---
  3. Scope

  In-Scope:
  - List all resumes: Show every resume belonging to the logged-in user with title, last-edited date, and status
  - Create new resume: Quick-create from dashboard → redirects to builder
  - View resume: Click resume from list → view-only rendered page
  - Edit resume: Click "Edit" → redirects to builder
  - Delete resume: Delete button with confirmation → removes from database
  - Set primary resume: Mark one resume as "primary" — one primary per user at all times
  - Duplicate resume: Copy all sections from one resume to a new draft
  - Resume status: "draft" (incomplete) or "complete"

  Out-of-Scope:
  - Bulk resume deletion
  - Resume versioning / version history
  - Sharing resumes via public link
  - Collaborative editing
  - Resume archiving (soft-delete)
  - Resume search/filter in list view
  - Custom folders

  ---
  4. User Stories

  US-P0.3-A: As a logged-in user, I want to see all my resumes on one page ordered by last-edited date, so that I can quickly find the resume I
  need.

  US-P0.3-B: As a logged-in user, I want to create a new blank resume with one click, so that I can start building a new resume immediately.

  US-P0.3-C: As a logged-in user, I want to edit an existing resume by clicking on it, so that I can update my information for new job
  applications.

  US-P0.3-D: As a logged-in user, I want to delete a resume I no longer need, so that I can keep my resume list organized and relevant.

  US-P0.3-E: As a logged-in user, I want to duplicate an existing resume as a starting point for a new one, so that I can create tailored
  resumes quickly without starting from scratch.

  US-P0.3-F: As a logged-in user, I want to mark one resume as my primary, so that it is pre-selected by default when applying for jobs (future
  integrations).

  P0.4: Resume Preview

  Primary Owners: Frontend Tech Lead, UI Component Agent
  Support: Wireframe Agent, CSS/Animation Agent, Template Architect Agent, QA Lead, Functional Test Agent, Browser Compatibility Agent
  Watchers: Product Manager Agent, System Architect

  1. Problem Statement

  Who has this problem? Every resume builder user who wants to know what their final resume will look like before exporting or submitting it.

  What is the problem? Without a live preview, users fill in forms blindly and only see the result when they export. If the formatting is wrong,
   sections are misaligned, or content overflows, they have to go back and re-edit — a frustrating cycle of trial and error.

  Why does it matter? The preview is the primary confidence-builder in the resume creation experience. Seeing a professional-looking output
  motivates users to complete their resumes. It also surfaces formatting issues early before the user invests time in a bad version.

  Current workaround (if any): Users export to PDF in other tools and review — but those tools don't have live preview, so the feedback loop is
  slow.

  ---
  2. Goal / Objective

  User Goal: See an accurate, real-time preview of my resume exactly as it will appear when exported — with clean typography, proper spacing,
  and professional layout.

  Business Goal: Increase resume completion rate by giving users visual feedback. Build a preview system that doubles as the print/export
  rendering engine (DRY — one template for both preview and PDF).

  Success Definition: The preview updates within 1 second of any form change. Three distinct template styles are available. The preview matches
  the exported PDF with pixel-level fidelity.

  ---
  3. Scope

  In-Scope:
  - Live preview panel: Embedded in the builder page (right panel), updating in real-time
  - Three template styles:
    - Classic: Traditional layout, serif fonts, standard margins, familiar format
    - Modern: Clean sans-serif, strong visual hierarchy, accent color on headers
    - Minimal: Maximum whitespace, simple typography, no decorative elements
  - Preview-to-print parity: Same template used for PDF export — no discrepancy
  - Responsive preview: Correct aspect ratio on desktop and tablet. Scaled-down on mobile.
  - Print-optimized CSS: @media print stylesheet hides navigation, sets margins, prevents page breaks inside section headers
  - View-only preview page: Full-page preview accessible at /resume/<id>/ — clean view without builder UI

  Out-of-Scope:
  - Custom template builder (drag-and-drop, color picker)
  - Template customization (fonts, colors, margins) — fixed styles per template
  - Watermark on preview
  - Preview comparison (side-by-side)
  - Zoom in/out controls
  - Dark mode preview

  ---
  4. User Stories

  US-P0.4-A: As a logged-in user building a resume, I want to see a live preview on the right panel, so that I can immediately see how my
  changes look in the final format.

  US-P0.4-B: As a logged-in user, I want to switch between three template styles (Classic, Modern, Minimal), so that I can choose the look that
  best fits my industry and personality.

  US-P0.4-C: As a logged-in user, I want to click "Preview" on a saved resume and see a clean, full-page view, so that I can review my resume

  US-P0.4-E: As a logged-in user, I want to print the resume directly from the preview (Ctrl/Cmd+P), so that I can get a physical copy without

  US-P0.4-F: As a logged-in user, I want my resume to not break across pages awkwardly when printed, so that it looks professional on paper.

  5. Page breaks avoided inside: section headers, job entries, education entries

  What is the problem? Users create resumes in the app but need them in a universally usable format (PDF) to email to recruiters, upload to job

  Why does it matter? PDF export is the primary deliverable of the entire product. If the export is broken, slow, or produces ugly output, the

  9. Dependencies & Assumptions

  Dependencies:
  - P0.4 (Resume Preview) — export reuses the preview template
  - A PDF generation library (see Technical Notes)
  - Paper size preference stored in user settings (P1.4)

  Assumptions:
  - WeasyPrint is the preferred PDF library because it renders HTML/CSS directly
  - Most users will export 1–2 page resumes
  - US Letter is the default paper size (primary user base is US college students/grads)
  - PDF generation happens server-side for consistent rendering
  - A 5-second timeout is reasonable on a standard web server

  Blockers if missing:
  - WeasyPrint must be installed and configured
  - Template rendering must be complete (preview must work before export)

  ---
  10. Technical Notes

  - PDF Library: Use WeasyPrint (pip install weasyprint) — renders HTML/CSS to PDF. Alternatives: pdfkit (wkhtmltopdf), xhtml2pdf (less
  reliable).
  - Export Endpoint: POST /resume/<id>/export/ — returns PDF as application/pdf with Content-Disposition: attachment;
  filename="John_Doe_Resume.pdf".
  - Rendering Pipeline:
    a. Load resume data from database
    b. Select template (user's saved preference)
    c. Render Django template to HTML string
    d. Pass HTML to WeasyPrint: HTML(string=html).write_pdf()
    e. Return PDF bytes as a FileResponse
  - Paper Size CSS: Use WeasyPrint's @page rule: @page { size: letter; margin: 0.5in; } or @page { size: A4; margin: 12.7mm; }.
  - Font Embedding: WeasyPrint bundles fonts. Use Google Fonts via local font cache or substitute with widely available system fonts.
  - Filename Sanitization: re.sub(r'[^\w\-]', '', name).replace(' ', '_') — remove non-ASCII from filename only.
  - Performance: Consider caching the rendered HTML string. Cache invalidates on any resume update.
  - Draft Warning: Frontend JavaScript checks resume.status === 'draft' before triggering export.
