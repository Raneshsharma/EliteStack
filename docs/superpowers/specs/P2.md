## P2.1: Landing Page

**Primary Owners:** `Frontend Tech Lead`, `UI Component Agent`
**Support:** `Landing Page Architect`, `Copywriter Agent`, `SEO Agent`, `Animation Specialist`, `QA Lead`
**Watchers:** `Product Manager Agent`, `Project Manager Agent`

### 1. Problem Statement

**Who has this problem?** Every visitor who discovers the product for the first time — through search engines, social media, word of mouth, or paid ads — and needs to understand what the product does, why it's better than alternatives, and whether they should sign up.

**What is the problem?** Without a landing page, unauthenticated visitors have no entry point to the product. They cannot understand the value proposition, see the product in action, or sign up without navigating to the signup page directly — which has high friction and low conversion.

**Why does it matter?** The landing page is the first impression of the product. It determines whether a visitor converts to a registered user or bounces. A well-designed landing page can 10x the signup conversion rate compared to sending visitors directly to the signup form. It is also the primary channel for organic (SEO) and paid acquisition.

**Current workaround (if any):** Without a landing page, visitors go to the signup page directly, which has no context, no social proof, and no explanation — leading to low conversion and high bounce rates.

---

### 2. Goal / Objective

**User Goal:** Understand what the product does, see how it works, and sign up in under 2 minutes — all without talking to anyone or reading extensive documentation.

**Business Goal:** Maximize visitor-to-registered-user conversion rate. Establish SEO presence for key search terms ("resume builder", "free resume maker"). Communicate the free-tier value proposition clearly to reduce acquisition cost.

**Success Definition:** The landing page loads in under 2 seconds. The hero section communicates the value proposition in under 5 seconds of reading. The signup conversion rate is above 5% of unique visitors. The page ranks for target keywords within 3 months of launch.

---

### 3. Scope

**In-Scope:**
- **Hero section:** Headline, subheadline, CTA button ("Create Your Free Resume"), and a live resume preview or screenshot
- **Features overview section:** 3–4 key features with icons and brief descriptions
- **How it works section:** 3-step visual flow (Create → Customize → Export)
- **Testimonials / social proof section:** 2–3 user testimonials with names, roles, and optional photos
- **Pricing tiers section:** Free / Pro / Premium tier comparison table
- **FAQ section:** 5–6 common questions and answers
- **Sign up CTA section:** Final call-to-action before the footer
- **Footer:** Links (About, Privacy Policy, Terms of Service, Contact), social media icons, copyright
- **Navigation:** Logo, Features, Pricing, FAQ links + "Log In" and "Sign Up" buttons
- **SEO metadata:** Page title, meta description, Open Graph tags, basic structured data
- **Responsive design:** Fully functional on desktop, tablet, and mobile
- **Performance:** Page load time under 2 seconds on a 3G connection

**Out-of-Scope:**
- Blog or content marketing hub
- Demo video or interactive product tour
- User account creation on the landing page (redirect to `/accounts/signup/`)
- A/B testing framework or experimentation platform
- Multi-language support or localization
- Affiliate program integration
- Press / media kit page
- Animated scroll effects or complex interactions beyond basic CSS
- Dark mode on landing page
- Cookie consent banner (can be deferred to P3)

---

### 4. User Stories

**US-P2.1-A:** As a **visitor who has never heard of the product**, I want to **immediately understand what the product does** from the hero section, so that I can **decide within 5 seconds if this is relevant to me**.

**US-P2.1-B:** As a **visitor**, I want to **see a preview of the product** (a sample resume), so that I can **visualize the end result before signing up**.

**US-P2.1-C:** As a **visitor**, I want to **understand the key features and benefits**, so that I can **compare this product to alternatives**.

**US-P2.1-D:** As a **visitor on a budget**, I want to **see clear pricing tiers** including a free option, so that I can **start using the product without any financial commitment**.

**US-P2.1-E:** As a **visitor with questions**, I want to **read an FAQ section**, so that I can **get answers without contacting support**.

**US-P2.1-F:** As a **visitor**, I want to **sign up in one click** from the landing page, so that I can **start using the product without friction**.

**US-P2.1-G:** As a **visitor who finds the product via Google**, I want the **landing page to be the top search result**, so that I can **easily find and access the product**.

### 5. User Flow / Experience

**Landing Page Scroll Flow:**
1. Visitor lands on the homepage (`/`)
2. Hero section is immediately visible above the fold
3. Hero shows: headline ("Build Professional Resumes in Minutes — Free"), subheadline, "Create Your Free Resume" CTA, sample resume screenshot or live preview
4. Visitor scrolls down → Features section → How It Works section → Testimonials → Pricing → FAQ → Final CTA → Footer
5. Navigation bar is sticky — always visible at the top while scrolling
6. "Sign Up" button in nav → redirects to `/accounts/signup/`
7. "Log In" button in nav → redirects to `/accounts/login/`

**Pricing Section Interaction:**
1. Three pricing cards: Free, Pro ($9.99/mo), Premium ($19.99/mo)
2. Each card shows: tier name, price, feature list, CTA button
3. Free tier CTA: "Get Started Free"
4. Pro/Premium CTA: "Upgrade to Pro" / "Upgrade to Premium" (links to signup with upgrade prompt)
5. "Most Popular" badge on the Pro tier
6. Annual billing toggle (optional): shows discounted monthly price when toggled on

**FAQ Section Interaction:**
1. FAQ items are collapsed by default (accordion style)
2. Click on a question → expands to show the answer
3. Only one FAQ item open at a time

**Mobile Flow:**
1. Hero section stacks vertically: headline, subheadline, CTA, preview
2. Navigation collapses to hamburger menu
3. Pricing cards stack vertically
4. All sections are single-column

### 7. Edge Cases & Error Handling

| Scenario | Expected Behavior |
|----------|-------------------|
| Landing page images fail to load | All images have `alt` text. Decorative images have fallbacks. No broken image icons shown. |
| JavaScript is disabled | Page renders fully via server-side templates. Accordions and mobile menu work via CSS-only fallback. |
| Landing page accessed on very small screen (320px) | Page is usable. Typography scales. CTA buttons are tappable (min 44px height). |
| Visitor clicks "Sign Up" multiple times rapidly | Only one redirect to `/accounts/signup/` per click. No duplicate tabs. |
| Pricing tier has a feature that doesn't exist yet | Grey out the feature with "(Coming Soon)" label instead of showing it as available |
| Testimonial has special characters (é, ñ, etc.) | All text is UTF-8 encoded. No broken characters shown. |
| Page is accessed via an old cached version | Proper cache-control headers prevent serving stale content for more than 1 hour |
| Very long testimonial text | Truncate at 150 characters with "..." and a "Read more" link |
| SEO crawlers (Googlebot) access the page | Page renders server-side (no JS dependency for content). All text is crawlable. |
| Dark mode preference detected | Landing page remains in light mode (no dark mode at MVP) |

### 10. Technical Notes

- **URL:** `GET /` — the root URL. For unauthenticated users, render the landing page template. For authenticated users, redirect to `/dashboard/`.
- **Template:** `templates/pages/landing.html` — extends `base.html`. All sections are template partials rendered server-side.
- **Static Files:** Tailwind CSS compiled to `static/css/output.css`. No JavaScript framework needed — vanilla JS for accordion and mobile menu.
- **Images:** Use `srcset` for responsive images. Host on AWS S3 or Cloudflare Images for performance. Placeholder: `https://via.placeholder.com/` or similar.
- **SEO:** Use Django's `django-meta` or manual template tags for `<title>`, `<meta name="description">`, Open Graph (`og:title`, `og:description`, `og:image`, `og:url`).
- **Structured Data:** Add `Organization` and `WebApplication` JSON-LD schema markup in the `<head>` for Google rich results.
- **Pricing Toggle (Annual):** If implemented, use a simple JavaScript toggle that shows/hides annual prices. Store preference in `localStorage`.
- **Performance:** Preload critical fonts. Defer non-critical CSS. Use `loading="lazy"` on below-fold images. Target Lighthouse score > 90.
- **Auth Redirect:** In the root view, check `request.user.is_authenticated`. If True, `return redirect('/dashboard/')`. If False, `return render(request, 'pages/landing.html')`.
- **Analytics:** Add Google Analytics 4 (or plausible analytics) tracking script. Track: page views, CTA clicks, scroll depth, signup completions.

## P2.2: Admin Panel

**Primary Owner:** `Backend Tech Lead`
**Support:** `Database Agent`, `Admin Config Agent`, `Security Test Agent`
**Watchers:** `System Architect`

### 1. Problem Statement

**Who has this problem?** The product team, developers, and anyone managing the platform — who needs visibility into user data, content moderation, and system health.

**What is the problem?** Without an admin panel, the team has no way to view user accounts, inspect resume data, moderate content, or troubleshoot issues without directly querying the database — which is slow, error-prone, and requires technical access.

**Why does it matter?** An admin panel enables the team to respond to support tickets quickly, identify abuse or spam, monitor platform health, and manage the product without engineering involvement for every operational task. It is a basic requirement for any production system.

**Current workaround (if any):** Engineers run SQL queries directly on the production database to look up user data or diagnose issues — a security risk and a bottleneck.

---

### 2. Goal / Objective

**User Goal:** As an admin, I want to browse, search, and manage all users and their resume data from a web interface — without writing SQL or accessing the server directly.

**Business Goal:** Enable the support team to resolve user tickets in under 5 minutes. Detect and address abuse. Monitor key platform metrics at a glance. Prepare the admin infrastructure for future scaling.

**Success Definition:** Django admin is fully configured with all models registered. Staff users can log in at `/admin/`. User management, resume browsing, and search work correctly. No sensitive data is exposed to non-staff users.

---

### 3. Scope

**In-Scope:**
- **Django admin configuration:** All models registered with appropriate admin classes
- **User management:** List, search, view, and disable user accounts
- **Resume data browsing:** View all resumes and their sections without editing
- **Resume content moderation:** Ability to soft-delete resumes that violate policies
- **Search and filtering:** Search users by email/username; filter resumes by status
- **Export data capability:** Export user list or resume data to CSV
- **Staff access control:** Only Django `is_staff=True` users can access `/admin/`
- **Admin dashboard:** Overview of platform statistics (total users, total resumes, signups this week)

**Out-of-Scope:**
- Role-based admin permissions (superuser vs. support vs. viewer — beyond basic staff flag at MVP)
- Content flagging / user reporting system
- Automated abuse detection
- Admin audit logs (who accessed what — add in P3 if needed)
- Billing management in admin (future — Stripe dashboard)
- Custom admin views beyond Django's built-in CRUD
- Email sending to users from admin
- White-label / multi-tenant admin

---

### 4. User Stories

**US-P2.2-A:** As an **admin (staff user)**, I want to **log in to the Django admin panel**, so that I can **manage the platform's data**.

**US-P2.2-B:** As an **admin**, I want to **search for users by email or username**, so that I can **quickly find the right account when handling a support ticket**.

**US-P2.2-C:** As an **admin**, I want to **view a user's resumes and their content**, so that I can **verify the quality of content or investigate a report**.

**US-P2.2-D:** As an **admin**, I want to **disable or delete user accounts**, so that I can **take action against abuse or comply with data deletion requests (GDPR)**.

**US-P2.2-E:** As an **admin**, I want to **export user data to CSV**, so that I can **run offline analysis or generate reports**.

**US-P2.2-F:** As an **admin**, I want to **see an overview dashboard** with key platform metrics, so that I can **monitor the health of the platform at a glance**.

**US-P2.2-G:** As a **non-staff user**, I want to be **denied access to `/admin/`**, so that **platform data is protected from regular users**.

---

### 5. User Flow / Experience

**Admin Login Flow:**
1. Admin navigates to `/admin/`
2. Django admin login page displayed
3. Staff user enters credentials and logs in
4. Admin dashboard loads showing: total users, total resumes, signups this week, resumes created this week
5. Sidebar shows: Users, Resumes, User Profiles, Auth (Groups, Users)

**User Management Flow:**
1. Admin clicks "Users" in the sidebar
2. User list displays: username, email, date joined, staff status, is active status
3. Admin searches for a user by typing in the search bar
4. Admin clicks on a user → user detail page shows: profile info, linked resumes, account status
5. Admin can toggle "Is Active" to disable the account
6. Admin clicks "Delete" to remove the user (with confirmation)

**Resume Browsing Flow:**
1. Admin clicks "Resumes" in the sidebar
2. Resume list displays: title, user, status, created date, updated date
3. Admin filters by status (draft/complete) or searches by resume title
4. Admin clicks on a resume → resume detail page shows all sections (Personal Info, Education, Experience, Projects, Skills)
5. Admin can delete the resume (soft-delete with confirmation)

**Data Export Flow:**
1. Admin navigates to the User list or Resume list
2. Admin clicks "Export to CSV" button (custom admin action)
3. CSV file is downloaded with the current list's data

**Non-Staff Access Flow:**
1. Non-staff user navigates to `/admin/`
2. Redirected to login page
3. If login succeeds with non-staff credentials → redirected to homepage with access denied message
4. Staff credentials → admin panel loads

### 7. Edge Cases & Error Handling

| Scenario | Expected Behavior |
|----------|-------------------|
| Admin tries to delete their own account | Block deletion with message: "You cannot delete your own account." |
| Admin tries to deactivate the only staff account | Block with message: "Cannot deactivate the only staff account." |
| Large dataset (100k+ users) in admin list | Django admin pagination at 100 per page. Search is indexed. |
| Resume has 500+ section entries | Admin detail page paginates related entries at 100 per page. |
| User with deleted resume accesses their builder | 404 page. Resume is gone. No error. |
| Admin changes a user's email to an already-taken email | Django admin validation prevents duplicate email. |
| Admin password is forgotten | Use Django's password reset via CLI (`python manage.py shell`) or server access. |
| Sensitive data visible in admin list | Exclude sensitive fields (password hash) from list display. Use `list_display` carefully. |
| CSRF attack on admin actions | Django admin has built-in CSRF protection. No custom handling needed. |
| User requests GDPR data deletion | Admin finds the user, exports their data to CSV, then deletes the account. |

### 10. Technical Notes

- **Django Admin Setup:** Register all models in `admin.py`. Use `admin.ModelAdmin` subclasses with `list_display`, `list_filter`, `search_fields`, `readonly_fields`, and `actions`.
- **User Admin:** `list_display = ('username', 'email', 'date_joined', 'is_staff', 'is_active')`. `search_fields = ('username', 'email')`. `list_filter = ('is_staff', 'is_active', 'date_joined')`.
- **Resume Admin:** `list_display = ('title', 'user', 'status', 'created_at', 'updated_at')`. `search_fields = ('title', 'user__username', 'user__email')`. `list_filter = ('status', 'created_at')`. `raw_id_fields = ('user',)` to prevent loading all users in a dropdown.
- **Inline Admin:** Use `admin.TabularInline` for ResumeEducation, ResumeExperience, ResumeProject, ResumeSkill inside the Resume admin page.
- **Custom Admin Action (CSV Export):** Create a custom `export_to_csv` action in `admin.py`. Use Python's `csv` module and Django's `HttpResponse` with `Content-Type: text/csv`.
- **Dashboard Stats:** Create a custom `index.html` admin template override or use `admin_views` to compute stats on the dashboard.
- **Security:**
  - `is_staff` required for admin access
  - `is_superuser` required for user deletion (optional restriction)
  - All admin actions are logged by Django's admin itself
  - Rate-limit the admin login page if possible (consider `django-axes` package if brute-force becomes an issue)
- **GDPR:** Add a custom admin action to export user data as JSON before deletion. Log deletion events for audit trail.
- **Pagination:** Django admin defaults to 100 per page. Adjust with `list_per_page = 100`.
